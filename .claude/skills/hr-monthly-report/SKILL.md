# CMoney HR 月報・管理決策視圖（產生標準）

> 定位（2026-06 決策）：月報＝「人才密度 OS」數據模型的**月度視圖**；目的不是報告「發生了什麼」，而是指出「**接下來該管什麼**」。
> 由 Agent（MCP 拉數據）或主管手動套用；去識別與 fail-safe 為強制護欄。

## 一、四層結構（決策優先，每月新增 # YYYYMMDD 區塊）
```
# YYYYMMDD ｜ HR 管理決策視圖

【1 決策層】本期需主管拍板/關注（頂部）
  每項 = 管理問題 + 關鍵數據 + 建議行動 + 不行動的代價

【2 預警層】紅綠燈
  關鍵人才流失風險、P0/P1 補缺延遲、人才密度下滑

【3 健康度層】
  招募漏斗瓶頸（開缺→pipeline→面試→offer→報到）、time-to-fill 紅綠燈、pipeline 覆蓋率

【4 密度層】（接人才密度 OS）
  人才密度指數、各事業單位 heatmap、壞的離職（優秀人才流失）趨勢

（附錄）OKR/KR 活動明細、補缺清單 — 保留供展開，非主軸
```
> 原 OKR/KR 內容不刪，**折進健康度層與密度層**，細目入附錄；主軸前移到「決策 → 預警」。

## 二、資料來源對應（Agent 拉數據用）
| 層 | 資料來源 |
|---|---|
| 決策/預警 | 由下列數據經規則衍生（流失風險、補缺延遲、密度下滑門檻） |
| 健康度 | 招募系統、缺額表、JIRA（開缺/補缺/pipeline/time-to-fill） |
| 密度 | 人力異動 DB、人才盤點表（進用品質、留任、壞的離職） |
| 附錄明細 | MA Program、Hunter 回報、活動排程、部落格交稿表 |

## 三、敏感資料去識別規則（強制）
- 離職名單/人才盤點/「壞的離職」→ 去識別（代號/職能代稱）或寫入權限受限區。
- 寫入的 Google Doc 僅限主管/HR 權限可見。
- 資料缺漏/異常 → 標「待補/異常」，**不杜撰**，並觸發警示（fail-safe）。

## 四、機制
- **角色分流視圖**：CEO 看密度趨勢與風險、事業主管看自己 BU、HR 看流程瓶頸（同數據、不同切面與權限）。
- **AI 主動推播**：不只等人讀月報，agent 推「本週你該關注的 3 件事」給各主管。

## 五、格式與語氣
- 繁體中文、客觀；**每個數據點都要能回答「所以主管該做什麼？」**
- 沿用「# 日期」分段，新增當月區塊、不覆蓋歷史；可比指標標出變化。

## 六、產出
輸出四層結構的當月視圖（Markdown）。Agent 模式經 Google Docs MCP 寫入；手動模式複製貼上。**月報、儀表板、預警推播皆為同一「人才密度 OS」數據模型的不同 render。**
