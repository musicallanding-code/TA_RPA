# CMoney HR 月報・六塊結構（產生標準）

> 定位（2026-06 決策）：月報＝「人才密度 OS」數據模型的**月度視圖**；目的是指出「**接下來該管什麼**」。
> 設計原則：**忠於月會 4 期既有欄目（OKR/KR1-3/People Operations）**，只做三個升級——
> ① 最前面加「決策摘要」 ② 每塊加「一句話狀態＋🔴🟡🟢」、明細收折 ③ 固定欄目模板化、敏感去識別。
> 護欄：去識別、fail-safe（缺資料標「待補」不杜撰）為強制。

## 一、六塊結構（每月新增 `# YYYYMMDD` 區塊）

```
# YYYYMMDD ｜ HR 月報

【0 本月決策摘要】需主管拍板/關注 3–5 件
  每項 = 問題 + 關鍵數據 + 建議行動 + 不行動的代價
  （把最重要的頂上來：壞的離職、報到率偏低、P0 補缺延遲…）

【1 People Operations / 人力狀態】  〔狀態一句話 + 🔴🟡🟢〕
  編制現況、人力異動（近 12 月到離職率/淨變化/趨勢）、人才盤點、壞的離職（優秀人才流失）
  ※ 敏感人事去識別（代號/職能代稱）、權限分層

【2 KR1 招募管道】  〔狀態一句話 + 🔴🟡🟢〕
  招募活動類（MA 等）、獵頭（護城河）、招募專案、各部門 P0/P1
  ※ 以漏斗＋紅綠燈呈現「補得順不順、卡在哪」；補缺名單收折

【3 KR2 雇主品牌】  〔狀態一句話 + 🔴🟡🟢〕
  雇主品牌活動（C Talk+、博覽會…）、部落格內容產製效率與影響力

【4 KR3 流程與系統】  〔狀態一句話 + 🔴🟡🟢〕
  招募現況、RSP 儀表板、內部推薦政策優化、AI 工具應用、人才吸引力調查

【5 下階段重點與風險】
  既有「下階段推進重點」，並回連【0 決策摘要】
```

## 二、資料來源對應（Agent 自動填）
| 塊 | 資料來源 |
|---|---|
| 0 決策摘要 | 由下列數據經規則衍生（壞的離職、報到率、補缺延遲門檻） |
| 1 人力狀態 | 人力異動 DB、人才盤點表、編制表 |
| 2 招募管道 | 招募系統、缺額表、JIRA、Hunter 回報、MA 數據 |
| 3 雇主品牌 | 活動排程、部落格交稿表 |
| 4 流程與系統 | RSP/缺額表儀表板、內推政策、AI 工具紀錄、人才吸引力調查 |

## 三、敏感資料去識別規則（強制）
- 離職名單/人才盤點/「壞的離職」→ 去識別（代號/職能）或寫入權限受限區。
- 寫入的 Google Doc 僅限主管/HR 權限可見。
- 資料缺漏/異常 → 標「待補/異常」、**不杜撰**、觸發警示（fail-safe）。

## 四、呈現方式
- **每塊頂部「一句話狀態 ＋ 🔴🟡🟢」**，明細收折/附錄——主管 30 秒掃完、要細節再展開。
- 數字標**可比變化**（如推薦數 11→20、報到率 58% vs 目標）。
- 每個數據點都要能回答「**所以主管該做什麼？**」

## 五、機制
- **角色分流視圖**：CEO 看人力狀態與風險、事業主管看自己 BU 招募/流失、HR 看流程瓶頸。
- **AI 主動推播**：除月報外，agent 推「本週你該關注的 3 件事」給各主管。

## 六、產出
輸出六塊結構的當月視圖（Markdown）。Agent 模式經 Google Docs MCP 寫入當月區塊（不覆蓋歷史）；手動模式複製貼上。**月報、儀表板、每週推播皆為同一「人才密度 OS」數據模型的不同 render。**
